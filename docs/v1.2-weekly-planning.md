# V1.2 Weekly Meal Planning

## Problem Statement

The app already helps pick a single dinner (random picker, ingredient matching, favorites). But every night the family still faces "what's for dinner?" because there is no way to plan ahead. The user opens the app, picks a meal, then does it again tomorrow — seven micro-decisions per week with no memory of the plan.

A lightweight weekly planner would let the family sit down once (e.g. Sunday), sketch out the week's dinners, and generate a single consolidated shopping list. Nightly dinner becomes "check the plan" instead of "make a decision."

## User Flow

1. **Create a plan.** The user opens a new "Weekly plan" panel, picks a week-start date (defaults to the upcoming Monday), and sees seven day slots (Mon–Sun).
2. **Fill slots.** For each day, the user can:
   - Search/browse and pick a meal from their library.
   - Use "Fill random" to auto-pick a meal for that slot (respects existing random-picker filters: favorites-only, pantry-ready, avoid-recently-served).
   - Leave a slot empty (no obligation to plan all seven days).
   - Clear a filled slot.
3. **Review.** The panel shows the filled plan at a glance — meal names, favorite badges, any notes.
4. **Generate shopping list.** One button generates a shopping list for all planned meals using the existing `POST /api/shopping-list/generate` endpoint — no new shopping-list logic needed.
5. **Commit plan.** When the user is happy, they save the plan. Saved plans are persisted so the family can revisit them mid-week.
6. **Serve from plan.** During the week, the user can mark a planned meal as "served," which logs to `meal_history` with source `plan`.

Only one active (current/future) plan exists at a time. Old plans are kept for reference but are not editable.

## In Scope

- New `weekly_plans` and `weekly_plan_slots` tables.
- CRUD API for weekly plans (create, get current, update slots, mark served).
- Frontend panel to build, view, and interact with the weekly plan.
- "Fill random" per-slot that reuses `suggestions.service.pickRandomMeal` logic, extended with an `excludeMealIds` filter (cross-module change to the suggestions service).
- "Generate shopping list" button that feeds all planned meal IDs into the existing shopping-list endpoint.
- Logging served meals to history with `source: 'plan'`.
- A single active plan constraint (one non-archived plan at a time).

## Out of Scope

- Multi-week planning or calendar views beyond a single 7-day plan.
- Drag-and-drop slot reordering.
- Automatic pantry depletion after serving.
- Meal prep/batch cooking grouping.
- Recurring/template plans ("copy last week").
- Notification or reminder system.
- Google Docs/export integration (but `copyText` from shopping list still works).
- Auth/multi-user.

## Data Model

### New table: `weekly_plans`

| Column        | Type    | Notes                                              |
|---------------|---------|----------------------------------------------------|
| `id`          | INTEGER | Primary key, autoincrement.                        |
| `week_start`  | TEXT    | ISO date (YYYY-MM-DD) of Monday. Unique among non-archived plans. |
| `is_archived` | INTEGER | Default 0. Set to 1 when a new plan supersedes it. |
| `created_at`  | TEXT    | Default `CURRENT_TIMESTAMP`.                       |
| `updated_at`  | TEXT    | Default `CURRENT_TIMESTAMP`.                       |

### New table: `weekly_plan_slots`

| Column    | Type    | Notes                                                    |
|-----------|---------|----------------------------------------------------------|
| `plan_id` | INTEGER | FK → `weekly_plans.id` ON DELETE CASCADE.                |
| `day`     | INTEGER | 0 = Monday, 1 = Tuesday, … 6 = Sunday.                  |
| `meal_id` | INTEGER | FK → `meals.id` ON DELETE SET NULL. Nullable (empty slot). |
| `notes`   | TEXT    | Nullable. Per-slot note, e.g. "Use leftover chicken."    |

Primary key: `(plan_id, day)`.

### Why this shape

- **7 fixed slots** rather than a flexible list keeps the domain simple and maps 1:1 to a visual week grid. No sort-order ambiguity.
- **`day` as integer 0–6** avoids date math per slot — the actual date is `week_start + day`.
- **Nullable `meal_id`** allows partial plans without placeholder meals.
- **`ON DELETE SET NULL`** on `meal_id` is a safety net for the rare case of a hard row deletion in the `meals` table. In normal operation meals are soft-archived (`is_archived = 1`), not deleted, so the FK stays intact. Plan slot queries should JOIN to `meals` and include `is_archived` so the API can return `isArchived: true` on the meal summary — the UI shows the planned meal name and indicates it is no longer active rather than silently dropping it.
- **No `served` column on slots.** Serving is recorded in `meal_history` (existing table) with `source = 'plan'`. We can join to check if a slot's meal was served on the slot's date.
- **`updated_at` on `weekly_plans`** is managed by application code, not a child-table trigger. Every repo function that mutates `weekly_plan_slots` must explicitly `UPDATE weekly_plans SET updated_at = CURRENT_TIMESTAMP WHERE id = ?` in the same transaction. The migration should still include a row-level trigger for direct updates to `weekly_plans` columns (matching the `trg_meals_updated_at` pattern), but slot mutations go through the repo path.

### Migration file

New file: `src/db/migrations/002_weekly_plans.sql`

Follows the existing pattern in `src/db/migrate.js` — the migration runner auto-discovers numbered `.sql` files.

### Interaction with existing tables

- **`meals`**: plans reference meals by ID. No changes to `meals` table.
- **`meal_history`**: serving from a plan writes a row with `source = 'plan'` and `served_on` = the slot's actual date. The existing `historyWriteSchema` already supports `source` as an enum — needs `'plan'` added to the allowed values.
- **`pantry_items`**: no changes. Shopping list generation already uses the pantry.
- **`ingredients` / `meal_ingredients`**: no changes. Shopping list generation already resolves ingredients through meals.

## API Design

### `POST /api/weekly-plans`

Create a new plan for a given week.

**Request:**
```json
{
  "weekStart": "2026-04-06"
}
```

**Rules:**
- `weekStart` must be a valid ISO date that falls on a Monday.
- If a non-archived plan already exists for that `weekStart`, return `409 Conflict`.
- Auto-archives any existing non-archived plan for a *different* week (single active plan policy).
- Creates the plan row plus 7 empty slot rows.

**Response:** `201` with the full plan object (see GET shape below).

### `GET /api/weekly-plans/current`

Get the single active (non-archived) plan.

**Response:** `200` with the plan object, or `404` if no active plan exists.

**Plan response shape:**
```json
{
  "data": {
    "id": 1,
    "weekStart": "2026-04-06",
    "isArchived": false,
    "createdAt": "2026-04-01T12:00:00",
    "updatedAt": "2026-04-01T12:00:00",
    "slots": [
      {
        "day": 0,
        "date": "2026-04-06",
        "label": "Monday",
        "meal": null,
        "notes": null,
        "served": false
      },
      {
        "day": 1,
        "date": "2026-04-07",
        "label": "Tuesday",
        "meal": { "id": 3, "name": "Spaghetti Tacos", "isFavorite": true, "isArchived": false },
        "notes": null,
        "served": true
      }
    ]
  }
}
```

All responses use the existing `{ data: ... }` envelope (see `src/modules/shopping-list/shopping-list.routes.js`, `src/modules/meals/meals.routes.js` for the pattern).

- `meal` is a summary object (`id`, `name`, `isFavorite`, `isArchived`) or `null`. When a meal has been soft-archived, the summary is still returned with `isArchived: true` so the UI can display the name and flag it.
- `served` is derived by checking `meal_history` for a matching `(meal_id, served_on)` where the date equals the slot's computed date. **Design note:** this derivation can false-positive if the same meal is logged to history on the same date via a different source (e.g. manual). This is acceptable for a single-household app; a future `plan_slot_id` FK on `meal_history` could disambiguate if needed. Implementers should add a code comment at the derivation query explaining this tradeoff.
- `date` and `label` are computed from `weekStart + day`.

### `PATCH /api/weekly-plans/current/slots/:day`

Assign or clear a meal for a specific day.

**Request:**
```json
{
  "mealId": 5,
  "notes": "Use the leftover rice"
}
```

- `mealId` can be `null` to clear the slot.
- `notes` is optional.
- `day` param is 0–6.

**Response:** `200` with the updated full plan object.

### `POST /api/weekly-plans/current/slots/:day/random`

Fill a slot using the random picker.

**Request (optional filters):**
```json
{
  "favoritesOnly": false,
  "fullMatchOnly": false,
  "excludeServedWithinDays": 7,
  "excludePlannedMeals": true
}
```

- `excludePlannedMeals` (default `true`) removes meals already assigned to other slots in the current plan from the random candidate pool. This avoids duplicates within a week without making it a hard constraint.

**Response:** `200` with the updated full plan object.

### `POST /api/weekly-plans/current/slots/:day/serve`

Mark the planned meal as served. Writes to `meal_history` with `source = 'plan'` and `served_on` = slot date.

**Response:** `200` with the updated full plan object.

**Error:** `400` if the slot has no meal assigned.

### `GET /api/weekly-plans/history`

List past (archived) plans, most recent first.

**Query:** `?limit=10`

**Response:** `{ data: [...] }` — array of plan summary objects (id, weekStart, createdAt) without full slot expansion.

### `GET /api/weekly-plans/history/:id`

Get one saved weekly plan by ID with the full plan payload shape used by current-plan reads.

**Response:** `200` with `{ data: plan }`, or `404` if the plan does not exist.

**Note:** This route is intended for opening archived plan history detail. It can also return a current plan when given that ID.

## Interaction with Existing Features

### Favorites

- The slot-assignment UI should show favorite badges on meals in the picker.
- "Fill random" can optionally filter to favorites-only, matching existing `pickRandomMeal` behavior.

### Random Picker

- "Fill random" reuses `suggestions.service.pickRandomMeal`, but the current function signature (`pickRandomMeal(filters)` in `src/modules/suggestions/suggestions.service.js`) does **not** accept an exclude-list. The function filters by `excludeServedWithinDays`, `favoritesOnly`, and `fullMatchOnly` only.
- **Required cross-module change:** add an optional `excludeMealIds` (array of integers) to the `filters` parameter of `pickRandomMeal`. When provided, the function should filter those IDs out of the candidate list after the existing `recentMealIds` filter and before selection. This is a small, additive change — no existing callers are affected because the field is optional.
- The weekly-plans service calls `pickRandomMeal` with `excludeMealIds` set to the meal IDs already assigned to other slots in the current plan (from `repo.listPlannedMealIds`). The `excludePlannedMeals` request flag (default `true`) controls whether this exclusion is applied.

### History

- "Serve" from a plan writes to `meal_history` with `source: 'plan'`, using the slot's actual date as `served_on`.
- The `excludeServedWithinDays` filter on random picks naturally incorporates plan-served meals too, since they're in the same history table.
- The existing history panel shows plan-served meals alongside manual ones.

### Shopping List

- The plan panel has a "Generate shopping list" button that collects all non-null `mealIds` from the plan's slots and calls `POST /api/shopping-list/generate` with those IDs.
- This is a **frontend-only integration**: no new backend endpoint needed. The existing shopping-list endpoint already accepts an array of meal IDs.
- The existing shopping-list panel can coexist; the plan just provides a convenient shortcut to populate `mealIds`.

### Pantry

- No direct interaction. Pantry is consumed indirectly through the shopping list.

## Acceptance Criteria

### Backend

1. `POST /api/weekly-plans` creates a plan with 7 slots and returns `201`.
2. Creating a plan auto-archives any prior active plan for a different week.
3. Creating a duplicate plan for the same `weekStart` returns `409`.
4. `weekStart` must be a Monday; non-Monday dates return `400`.
5. `GET /api/weekly-plans/current` returns the active plan with all 7 slots, including meal summaries and `served` status.
6. `GET /api/weekly-plans/current` returns `404` when no active plan exists.
7. `PATCH .../slots/:day` assigns a meal or clears a slot (`mealId: null`).
8. `PATCH .../slots/:day` with an invalid meal ID returns `404`.
9. `POST .../slots/:day/random` fills the slot with a random meal, excluding meals already in other slots by default.
10. `POST .../slots/:day/serve` creates a `meal_history` entry with `source = 'plan'` and the correct `served_on` date.
11. `POST .../slots/:day/serve` returns `400` when the slot is empty.
12. If a meal is soft-archived, the slot still returns the meal summary with `isArchived: true`. If a meal row is hard-deleted (unusual), `ON DELETE SET NULL` clears the FK and the slot returns `meal: null`.
13. `GET /api/weekly-plans/history` lists archived plans.

### Frontend

1. A "Weekly plan" panel is visible and shows 7 day rows.
2. Each row shows the day label, date, assigned meal (or "Empty"), and action buttons.
3. User can pick a meal from a dropdown/search for each slot.
4. "Fill random" on a slot populates it and shows the result immediately.
5. "Clear" removes the meal from a slot.
6. "Generate shopping list" collects plan meals and triggers shopping-list generation; result appears in the existing shopping-list panel.
7. "Serve" button on a slot marks the meal as served and updates the visual state.
8. Creating a new plan for a new week is accessible (e.g. a "New week" button with date picker).
9. The panel handles the no-active-plan state with a prompt to create one.

## Risks and Assumptions

| Risk / Assumption | Impact | Mitigation |
|---|---|---|
| Single active plan is too restrictive for some families. | Low. Target user is one family, one week at a time. | The model supports archived plans; "copy from previous" can be added later without schema changes. |
| Soft-archived meals remain in plan slots (FK intact) but are no longer in the active meal list. | Low — the user still sees the meal name in the plan. | The meal summary in the plan response includes `isArchived: true`. Frontend should show a visual indicator (e.g. strikethrough or "(archived)" badge). The user can clear the slot and pick a replacement. |
| Deriving `served` by joining `meal_history` on `(meal_id, served_on)` could false-positive if the same meal was manually served on the same date outside the plan. | Low — single-household use and low data volume make this unlikely. | Acceptable for v1.2. A `plan_slot_id` FK on `meal_history` could disambiguate later. |
| The "Fill random" feature adds a new per-slot endpoint rather than reusing the existing `GET /api/suggestions/random`. | Low — keeps the plan module self-contained and lets us add `excludePlannedMeals` cleanly. | Reuse the `pickRandomMeal` service function internally (with the new `excludeMealIds` param); only the route is new. |
| Quantity-free shopping lists for a full week may look messy with many `quantityHints`. | Medium — UX concern, not a data problem. | The existing `copyText` format already handles multi-meal lists. Verify readability in manual testing with 5+ meals. |
