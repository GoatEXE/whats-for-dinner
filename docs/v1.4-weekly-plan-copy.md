# V1.4 Copy Weekly Plan as Plain Text

## Problem Statement

The weekly planner lets the family build a 7-day dinner plan, but there is no way to share it. If one parent plans the week and the other asks "what's for dinner tonight?", the answer is "open the app." A quick copy-to-clipboard action would let the user paste the plan into a text message, a note, or a Google Doc that the whole family can glance at — without needing to open the app.

The shopping list already has a successful copy-to-clipboard pattern (`public/app.js` → `copyShoppingList()` using `navigator.clipboard.writeText`). This feature applies the same pattern to the weekly plan itself.

## Current Repo Context

Inspected files relevant to this slice:

- **`public/app.js` → `state.currentPlan`** — the active plan is already fully loaded in client state with the shape `{ id, weekStart, slots: [{ day, date, label, meal: { id, name, isFavorite, isArchived } | null, notes, served }] }`. All data needed for copy text is present.
- **`public/app.js` → `copyShoppingList()`** (line ~779) — existing clipboard pattern: `navigator.clipboard.writeText(text)` with `showStatus()` for success/failure feedback.
- **`public/app.js` → `renderWeeklyPlan()`** (line ~831) — renders the plan panel including a `plan-footer` div with the existing "Generate shopping list from plan" button. The copy button belongs here.
- **`public/app.js` → `handleWeeklyPlanActions()`** (line ~1070) — dispatches plan-panel button clicks by `data-action`. The new button hooks in here.
- **`src/modules/shopping-list/shopping-list.service.js` → `formatShoppingListPlainText()`** — the shopping list copy text is generated server-side because it involves cross-meal ingredient aggregation and pantry comparison. A weekly plan copy has no such aggregation — it is a pure presentation of 7 slots — so a backend endpoint is not warranted.

## Decision: Frontend-Only

**The copy text should be generated entirely in the frontend.** Rationale:

1. **All data is already in `state.currentPlan`.** The slot objects include `label`, `date`, `meal.name`, `meal.isArchived`, `notes`, and `served` — everything needed for the output.
2. **No aggregation or cross-module logic is required.** Unlike the shopping list (which deduplicates ingredients across meals and compares against pantry), the plan copy is a linear walk over 7 slots.
3. **No new endpoint means no backend work package, no tests for a format endpoint, and no round-trip latency.** The button acts instantly.
4. **Precedent:** the shopping list `copyText` is backend-generated for good reason (aggregation). Putting trivial formatting on the backend just for consistency would add unnecessary API surface.

If a future feature needs server-rendered plan text (e.g. email or scheduled export), a backend formatter can be added then without breaking this.

## User Flow

1. User has an active weekly plan with at least one meal assigned.
2. User clicks a **"Copy plan"** button in the plan panel footer (next to the existing "Generate shopping list from plan" button).
3. The plan is formatted as plain text and copied to the clipboard.
4. The status banner shows "Weekly plan copied to clipboard" (or an error fallback).

That's it. One click, one copy, done.

## Copy Text Format

The output should be clean enough to paste into a text message, a Google Doc, or a note app without editing.

### Format specification

```
Dinner plan — week of Apr 6, 2026

Monday    Apr 6   Spaghetti Tacos ✓
Tuesday   Apr 7   Black Bean Quesadillas — use leftover rice
Wednesday Apr 8   (no plan)
Thursday  Apr 9   Chicken Stir-Fry
Friday    Apr 10  Grilled Salmon
Saturday  Apr 11  (no plan)
Sunday    Apr 12  Pizza Night ★
```

### Formatting rules

| Element | Representation | Rationale |
|---|---|---|
| **Header line** | `Dinner plan — week of {formatted weekStart}` | Identifies the plan at a glance. Use locale-friendly short date (e.g. "Apr 6, 2026"). |
| **Blank line** | After header | Visual separation. |
| **Day label** | Left-padded to 9 chars (longest is "Wednesday") | Aligns the columns for readability in monospace and proportional fonts. |
| **Date** | Short format without year, e.g. "Apr 6" | Compact, scannable. The year is already in the header. Use `toLocaleDateString` with `{ month: "short", day: "numeric" }` — a year-less variant of the existing `formatDateShort` helper. |
| **Meal name** | As-is from `slot.meal.name` | The core information. |
| **Empty slot** | `(no plan)` | Honest and concise. Not "—" (ambiguous) or omitted (confusing). |
| **Served marker** | `✓` after the meal name | Useful mid-week to show progress. Placed after the name, before notes. |
| **Favorite marker** | `★` after the meal name (after ✓ if both) | Nice-to-have context. Uses a plain star that renders in all text contexts. |
| **Archived meal** | `(archived)` after the meal name | Signals the meal is no longer in the active library. |
| **Slot notes** | `— {notes}` appended after markers | Preserves the user's context like "use leftover chicken." |
| **Trailing newline** | One at the end | Clean paste behavior. |

### Why this format

- **No bullet points or Markdown.** The plan is a short fixed-length list. Bullets add visual noise and imply an unordered set, but days have inherent order.
- **No indentation hierarchy.** Everything is on one line per day. Notes are appended, not nested.
- **Served/favorite/archived are short suffixes, not separate columns.** Keeps lines scannable.
- **The `(no plan)` placeholder is explicit.** When pasted, the reader sees "nothing planned for Wednesday" rather than wondering if the line was accidentally deleted.

### Edge case: completely empty plan

If no meals are assigned at all, the copy should still work and produce:

```
Dinner plan — week of Apr 6, 2026

Monday    Apr 6   (no plan)
Tuesday   Apr 7   (no plan)
Wednesday Apr 8   (no plan)
Thursday  Apr 9   (no plan)
Friday    Apr 10  (no plan)
Saturday  Apr 11  (no plan)
Sunday    Apr 12  (no plan)
```

This is preferable to blocking the copy action, since the user might want to share a partially-filled plan before it's complete.

## In Scope

- A `formatWeeklyPlanText(plan)` pure function in `public/app.js` that takes `state.currentPlan` and returns a plain-text string.
- A "Copy plan" button in the plan panel footer.
- Clipboard write via `navigator.clipboard.writeText` with `showStatus` feedback.
- Handling the no-active-plan case (button not rendered when `state.currentPlan` is null).

## Out of Scope

- Backend endpoint for plan copy text.
- HTML or rich-text clipboard format.
- Google Docs integration.
- Copy action for archived/past plans (can be added later by reusing the same formatter with `state.expandedPlanDetail`).
- Customizable copy format or user preferences for what to include.
- A preview `<pre>` block of the text before copying (the shopping list has one, but the plan is visible on screen already — a preview would be redundant).

## Interaction with Existing Features

### Shopping list copy

No conflict. The shopping list "Copy to clipboard" button copies ingredient data. The plan "Copy plan" button copies the dinner schedule. They serve different purposes and both live in their own panels.

### Plan notes

Slot notes are included in the copy output. The existing notes-editing flow (`savePlanSlotNotes` in `app.js`, line ~1109) persists notes via `PATCH /api/weekly-plans/current/slots/:day`. Unsaved note edits (still in the DOM input but not yet PATCH'd) are **not** included in the copy — the formatter reads from `state.currentPlan`, which reflects the last server-confirmed state. This is consistent with how the shopping list generates from server state, not DOM state.

### Served status

Included as `✓` in the copy text. Since served status is derived server-side (from `meal_history` joined by `meal_id` + `served_on`), it is already accurate in `state.currentPlan` as of the last load/refresh.

### Archived meals

Included with `(archived)` suffix. The user can see which planned meals are no longer in the active library and decide whether to swap them out.

### Past plans

The copy button is **only on the active plan** in this slice. Past plans can get a copy button later by calling the same `formatWeeklyPlanText` function with the expanded plan detail object — the data shape is identical (both come from `buildPlan` in the repo). Noted as a natural follow-up but out of scope to keep this slice minimal.

## Acceptance Criteria

1. When an active plan exists and has at least one slot visible, a "Copy plan" button appears in the plan panel footer area.
2. Clicking "Copy plan" copies the formatted plan text to the clipboard.
3. The status banner shows "Weekly plan copied to clipboard" on success.
4. If `navigator.clipboard.writeText` fails (e.g. non-HTTPS in some browsers), the status banner shows a fallback error message.
5. The copy text includes all 7 days, with meal names for filled slots and `(no plan)` for empty slots.
6. Served meals are marked with `✓`.
7. Favorite meals are marked with `★`.
8. Archived meals are marked with `(archived)`.
9. Slot notes are appended after markers with `— {notes}`.
10. The header line includes the plan's week-start date in a readable short format.
11. The button does not appear when no active plan exists.

## Risks and Assumptions

| Risk / Assumption | Impact | Mitigation |
|---|---|---|
| `navigator.clipboard.writeText` requires a secure context (HTTPS or localhost) in some browsers. | Low — the app runs locally via Docker on localhost. | The existing shopping-list copy already depends on this API and works. Same fallback error pattern. |
| Unsaved note edits in the DOM are not included in the copied text. | Low — notes auto-save on change via `savePlanSlotNotes`. | The `change` event handler persists notes immediately. The only window for stale data is if the user is mid-keystroke, which is an edge case not worth additional complexity. |
| Day-label padding assumes English labels. | None for current use. | Single-household English-language app. Localization is a future concern. |
| The plain-text format may not align perfectly in proportional fonts. | Low — the content is still scannable. | The format is designed to be readable in both monospace and proportional contexts. Alignment is nice-to-have, not critical. |

## Implementation Tasks

### Frontend-Only Work Package

**Scope:** `public/app.js`

**No backend changes. No migration. No new endpoints. No new test files required** (the formatter is a pure function that can be verified manually; if the team prefers, a lightweight unit test can be added later, but it's not blocking).

**Work:**

1. **Add `formatWeeklyPlanText(plan)` function:**
   - Takes a plan object (same shape as `state.currentPlan`).
   - Returns a plain-text string following the format specification above.
   - Implementation sketch:
     ```js
     function formatDateCompact(isoDate) {
       const [y, m, d] = isoDate.split("-").map(Number);
       return new Date(y, m - 1, d).toLocaleDateString(undefined, {
         month: "short",
         day: "numeric",
       });
     }

     function formatWeeklyPlanText(plan) {
       const weekLabel = formatDateShort(plan.weekStart); // existing helper, includes year
       const lines = [`Dinner plan — week of ${weekLabel}`, ""];

       for (const slot of plan.slots) {
         const dayPart = slot.label.padEnd(9);
         const datePart = formatDateCompact(slot.date); // no year — header covers it

         if (!slot.meal) {
           lines.push(`${dayPart} ${datePart}  (no plan)`);
           continue;
         }

         let mealPart = slot.meal.name;
         if (slot.served) mealPart += " ✓";
         if (slot.meal.isFavorite) mealPart += " ★";
         if (slot.meal.isArchived) mealPart += " (archived)";
         if (slot.notes) mealPart += ` — ${slot.notes}`;

         lines.push(`${dayPart} ${datePart}  ${mealPart}`);
       }

       return lines.join("\n") + "\n";
     }
     ```
   - Reuses the existing `formatDateShort(isoDate)` helper (line ~1146 in `app.js`) for the header line (returns locale-friendly dates like "Apr 6, 2026").
   - Adds a small `formatDateCompact(isoDate)` helper for per-day dates that omits the year (e.g. "Apr 6").

2. **Add `copyWeeklyPlan()` async function:**
   - Guards on `state.currentPlan` being non-null.
   - Calls `formatWeeklyPlanText(state.currentPlan)`.
   - Calls `navigator.clipboard.writeText(text)`.
   - Calls `showStatus("Weekly plan copied to clipboard")` on success.
   - Catches and calls `showStatus("Failed to copy — try selecting the text manually", "error")` on failure.
   - Follows the exact same pattern as `copyShoppingList()` (line ~779).

3. **Add "Copy plan" button to `renderWeeklyPlan()`:**
   - In the `plan-footer` div, add a secondary button next to the existing "Generate shopping list from plan" button:
     ```html
     <button type="button" data-action="plan-copy" class="secondary">Copy plan</button>
     ```
   - The footer (and both buttons) only render when `plannedMealCount > 0`. This is already the existing conditional — no change needed to the guard, just add the button inside it.
   - **Refinement:** also render the copy button when `plannedMealCount === 0` if the user might want to copy an empty-week plan. Recommendation: keep the existing guard (`plannedMealCount > 0`) for now — an all-empty plan has nothing worth sharing. This can be relaxed later trivially.

4. **Wire the action in `handleWeeklyPlanActions()`:**
   - Add a case for `action === "plan-copy"` that calls `copyWeeklyPlan()`.

**Dependencies:** None.

**Verification:** Manual testing:
- Create a plan with a mix of filled slots, empty slots, served meals, favorited meals, archived meals, and slot notes.
- Click "Copy plan" and paste into a text editor. Verify:
  - Header line with correct week date.
  - All 7 days present.
  - Empty slots show `(no plan)`.
  - Served meals have `✓`.
  - Favorites have `★`.
  - Archived meals have `(archived)`.
  - Notes are appended with `—`.
  - Status banner shows success.
- Test clipboard failure by temporarily overriding `navigator.clipboard` (or testing in an insecure context) to verify the error fallback.
