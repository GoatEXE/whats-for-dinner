# V1.1 Shopping List Feature Slice

## Current repo context
Inspected files relevant to this slice:
- `README.md` — confirms exact normalized ingredient matching and quantity text as display-only.
- `src/app.js` — current app wiring, JSON API routes, and static `public/` UI hosting.
- `src/db/migrations/001_init.sql` — `meal_ingredients.quantity_text` and `pantry_items.quantity_text` exist, but there is no quantity math model and no shopping-list tables.
- `src/modules/suggestions/suggestions.service.js` — already resolves available ingredients from pantry/ad hoc input and computes missing required ingredients.
- `src/modules/meals/meals.repo.js` — meal records already return ingredients with `ingredientId`, `name`, `quantityText`, and `isOptional`.
- `src/modules/pantry/pantry.repo.js` — pantry is stored as ingredient presence plus optional quantity text.
- `public/index.html`, `public/app.js`, `public/styles.css` — current frontend is a single vanilla JS page with no build step, so UI changes should stay lightweight.

## Goal
Generate a copy-friendly shopping list for one or more selected meals by comparing those meals against saved pantry items and optional one-off on-hand ingredients.

## Assumptions and ambiguities
- **Presence-based matching only.** Because quantities are free text in the current schema, v1.1 should treat an ingredient as either on-hand or not on-hand. No unit conversion or quantity subtraction.
- **Exact normalized matching is acceptable.** This matches current behavior in `src/lib/normalize.js` and the suggestions flow.
- **No persistence required.** The shopping list can be generated on demand and kept in client state only.
- **Ad hoc ingredient names may create ingredient records.** If implementation reuses the existing suggestions/catalog resolution path, unknown `ingredientNames` may be inserted into `ingredients`. That is already the current behavior for ad hoc match inputs and is acceptable for this slice unless we explicitly decide to change it later.
- **UI selection should only expose active meals.** Backend can stay permissive and generate from any valid meal ID, but the normal UI should use active meals from `GET /api/meals`.

## In scope
- Generate a shopping list from **selected meal IDs**.
- Compare selected meals against:
  - saved pantry items
  - optional ad hoc on-hand ingredients supplied at request time
- Deduplicate shared ingredients across multiple meals.
- Distinguish:
  - required items still needed
  - required items already on hand
  - optional items still needed
- Return a plain-text `copyText` output that pastes cleanly into a Google Doc later.
- Add a minimal UI flow in the existing vanilla frontend to select meals, generate the list, preview it, and copy it.

## Out of scope
- Google Docs or Google Drive integration
- Saving shopping lists in the database
- Persisted item-level checkboxes or “mark purchased” state
- Quantity math, unit conversion, or pantry depletion
- Fuzzy ingredient aliases/synonyms beyond current normalized exact matching
- Meal scaling by servings
- Auto-generation from match cards or random results in the first pass

## Recommended API
### Endpoint
`POST /api/shopping-list/generate`

### Request body
```json
{
  "mealIds": [1, 2],
  "useSavedPantry": true,
  "ingredientIds": [5, 8],
  "ingredientNames": ["Milk"],
  "includeOptional": false
}
```

### Request rules
- `mealIds` is required and must contain at least 1 meal ID.
- `mealIds` should be deduplicated server-side.
- `useSavedPantry` defaults to `true`.
- `ingredientIds` and `ingredientNames` are optional and **add to** the saved pantry set.
- If `useSavedPantry=false` and no ad hoc ingredients are supplied, the generator should treat available ingredients as empty and return the full required list.
- `includeOptional` defaults to `false`.

### Response shape
```json
{
  "selectedMeals": [
    { "id": 1, "name": "Spaghetti Tacos" },
    { "id": 2, "name": "Black Bean Quesadillas" }
  ],
  "availableIngredients": [
    { "ingredientId": 10, "name": "Ground beef" },
    { "ingredientId": 11, "name": "Milk" }
  ],
  "summary": {
    "selectedMealCount": 2,
    "requiredToBuyCount": 3,
    "requiredOnHandCount": 4,
    "optionalToBuyCount": 1
  },
  "requiredToBuy": [
    {
      "ingredientId": 22,
      "name": "Taco shells",
      "mealNames": ["Spaghetti Tacos"],
      "quantityHints": ["Spaghetti Tacos — 8 shells"]
    }
  ],
  "requiredOnHand": [
    {
      "ingredientId": 9,
      "name": "Ground beef",
      "mealNames": ["Spaghetti Tacos"],
      "quantityHints": ["Spaghetti Tacos — 1 lb"]
    }
  ],
  "optionalToBuy": [
    {
      "ingredientId": 35,
      "name": "Salsa",
      "mealNames": ["Black Bean Quesadillas"],
      "quantityHints": ["Black Bean Quesadillas — for serving"]
    }
  ],
  "copyText": "Shopping list for: Spaghetti Tacos, Black Bean Quesadillas\n\nNeed to buy:\n- Taco shells — Spaghetti Tacos (8 shells)\n\nOptional:\n- Salsa — Black Bean Quesadillas (for serving)"
}
```

## Aggregation rules
- Group ingredients by canonical ingredient identity (`ingredientId` / existing normalized ingredient).
- If an ingredient is required in **any** selected meal, treat it as **required** in the grouped output.
- If the ingredient is on hand, mark it as covered for all selected meals that use it.
- Do **not** sum or subtract quantities; instead, return `quantityHints` per meal.
- Optional ingredients should stay out of `requiredToBuy`.
- Sort grouped output alphabetically by ingredient name for stable copy/export behavior.

## Copy/export behavior
Return `copyText` from the backend so the format stays consistent across UI and future export options.

Recommended plain-text format:
- first line identifies selected meals
- blank line
- `Need to buy:` section with one ingredient per line
- optional blank line and `Optional:` section when requested and non-empty

Recommended line format:
- `- Ingredient name — Meal A (qty); Meal B (qty)`

This should be plain text, not HTML and not a table.

## UI behavior
Keep the UI inside the existing static page.

### Minimal frontend flow
1. Add an **“Add to shopping list” / “Remove from shopping list”** action to meal cards.
2. Add a new **Shopping list** panel showing:
   - selected meal chips/names
   - `Use saved pantry` checkbox, checked by default
   - optional textarea for extra on-hand ingredient names, one per line
   - `Include optional ingredients` checkbox
   - `Generate list` button
   - `Copy` button
3. After generation, render:
   - a structured preview of required-to-buy / on-hand / optional sections
   - a read-only plain-text area or `<pre>` showing `copyText`
4. Keep selected meals and generated result in frontend state only.

### UI notes
- Do not add a frontend framework or build step.
- Follow existing `public/app.js` state/event pattern and current CSS utility classes from `public/styles.css`.
- Reuse the status banner for success/error/copy feedback.

## Data and storage implications
- **No migration expected.** Existing tables are sufficient.
- **No new persistence required.** Generated lists are ephemeral.
- A new backend module is enough, for example `src/modules/shopping-list/` with route/schema/service files.
- To keep complexity low, this module can compose existing repos/services rather than introducing new tables or generalized abstractions.

## Acceptance criteria
### Backend
- `POST /api/shopping-list/generate` returns `200` for valid meal selections.
- When two selected meals share the same missing ingredient, that ingredient appears **once** in `requiredToBuy` with both meal references in `mealNames`/`quantityHints`.
- When an ingredient exists in pantry/on-hand inputs, it is excluded from `requiredToBuy` and included in `requiredOnHand`.
- Optional ingredients are excluded by default and included in `optionalToBuy` only when `includeOptional=true`.
- If `useSavedPantry=false` and no ad hoc ingredients are provided, all required ingredients across selected meals appear as needed.
- Invalid or missing meal IDs return an error using the existing JSON error shape.
- `copyText` is present, non-empty, plain text, and contains the selected meal names plus the required items to buy.

### Frontend
- A user can select one or more meals from the existing meals list and generate a shopping list without leaving the page.
- The generated list is visibly split into at least `Need to buy` and `Already on hand`; `Optional` appears only when relevant.
- The Copy button copies the backend-provided `copyText` and shows confirmation in the existing status banner.
- Clearing or changing the selection regenerates the result cleanly without page reload.
