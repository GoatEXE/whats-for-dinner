# V1.7 Tabbed Layout — Mobile-Friendly Information Architecture

**Status:** ✅ Complete (shipped 2026-04-07)

## Problem Statement

The app currently shows 8 panels on a single scrolling page: quick picker, weekly plan (with past plans), shopping list, pantry, ingredient matches, meal editor, meals library, and history. On desktop this is a long page; on a phone it is an overwhelming vertical scroll where the user must hunt for the section they need.

The user has asked for tabs to reduce clutter. The core insight is that the app serves three distinct modes of use:

1. **"What's for dinner this week?"** — planning and picking meals (the most frequent use)
2. **"What do we need to buy?"** — shopping list generation and pantry management
3. **"What meals do we have / add a new one?"** — managing the meal library

These modes are rarely interleaved. A tab structure that maps to these modes lets the user focus on one task at a time while keeping the full feature set one tap away.

## Current Repo Context

Inspected files:

- **`public/index.html`** — 263 lines. Eight `<section>` elements inside `<main class="layout">`. Each section is a panel with a header, content, and optional forms. No tab markup or navigation exists.
- **`public/styles.css`** — 496 lines. Layout uses `display: grid` with `grid-template-columns: repeat(auto-fit, minmax(320px, 1fr))`. Panels marked `.panel-wide` span full width. One `@media (max-width: 720px)` breakpoint adjusts padding and plan-slot grid.
- **`public/app.js`** — 1,653 lines. Rendering is function-per-panel (`renderMeals`, `renderWeeklyPlan`, `renderPantry`, etc.). State is global. `loadData()` fetches all data at init and re-renders all panels. Event delegation is per-container element (e.g., `elements.weeklyPlanContent.addEventListener`).
- **`docs/cleanup-backlog.md`** — item #6 notes the 1,650-line monolith and suggests eventual ES module split. Item #7 notes mobile plan-slot layout issues. Both are addressed by this restructuring.

## Proposed Tab Structure

### Three tabs

| Tab | Label | Contents | Rationale |
|---|---|---|---|
| **Plan** | 🗓 Plan | Quick picker, Weekly plan (with autofill, notes, serve, past plans), History | The "decide and track dinners" workflow. Everything about choosing and recording what to eat. |
| **Shop** | 🛒 Shop | Shopping list, Pantry, Ingredient matches | The "what do we need" workflow. Pantry and matches are tightly coupled to shopping — pantry drives match results, and matches inform what to buy. |
| **Meals** | 🍽 Meals | Meal editor (add/edit form), Meals library (card grid with search) | The "manage the catalog" workflow. Adding, editing, browsing, and archiving meals. |

### Why this grouping

- **Plan tab** is the default landing tab — it answers the nightly question "what's for dinner?" and is the highest-frequency use.
- **Shop tab** groups pantry management with shopping list because they serve the same trip: "I'm going to the store." Ingredient matches are included here because they answer "what can I cook with what I have?" — a pantry-adjacent question that informs shopping decisions.
- **Meals tab** isolates the catalog management work. Adding/editing meals and browsing the library are setup tasks that happen less often than planning or shopping.
- **History** lives on the Plan tab because it shows what was actually served — directly relevant context when planning the next week ("we had tacos recently"). It is rendered below the weekly plan and past plans.

### What about the shopping list's cross-tab interactions?

The weekly plan has a "Generate shopping list from plan" button and a "Copy plan & shopping list" share-pack button. These currently populate the shopping-list panel and scroll to it. With tabs, these actions should:
- Generate the shopping list in the background (API call, update `state.shoppingListResult`)
- Switch to the **Shop tab** automatically so the user sees the result
- Show the status banner confirmation (which is global, above tabs)

The "Add to list" button on meal cards (Meals tab) should similarly switch to the Shop tab after toggling the meal. This cross-tab navigation is the only inter-tab interaction and is handled by programmatic tab switching.

## Global Elements (Outside Tabs)

These stay visible regardless of the active tab:

| Element | Reason |
|---|---|
| **Page header** (title, subtitle, Refresh button) | App identity and global reload. |
| **Status banner** | Feedback for all actions across all tabs. Must remain visible/announced after tab switches. |

## Tab Navigation Design

### HTML structure

```html
<nav class="tab-bar" aria-label="Main navigation">
  <button class="tab-button active" data-tab="plan" aria-selected="true" role="tab">🗓 Plan</button>
  <button class="tab-button" data-tab="shop" aria-selected="false" role="tab">🛒 Shop</button>
  <button class="tab-button" data-tab="meals" aria-selected="false" role="tab">🍽 Meals</button>
</nav>

<main>
  <div class="tab-panel active" id="tab-plan" role="tabpanel" aria-labelledby="...">
    <!-- Quick picker, Weekly plan, History -->
  </div>
  <div class="tab-panel" id="tab-shop" role="tabpanel" aria-labelledby="...">
    <!-- Shopping list, Pantry, Ingredient matches -->
  </div>
  <div class="tab-panel" id="tab-meals" role="tabpanel" aria-labelledby="...">
    <!-- Meal editor, Meals library -->
  </div>
</main>
```

### Tab switching behavior

- Only one tab panel is visible at a time. Inactive panels use `display: none` (not removed from DOM — preserves form state, scroll position, and avoids re-rendering).
- Switching tabs updates `aria-selected` on buttons and toggles `.active` on panels.
- The active tab is stored in `state.activeTab` and persisted to `sessionStorage` so a page refresh returns to the same tab (not `localStorage` — this is a session-level preference, not permanent).
- No URL hash routing. The app is a single page and doesn't need bookmarkable tab URLs.

### Mobile behavior

- The tab bar is a fixed row at the **bottom** of the screen on mobile (`position: sticky; bottom: 0`), matching native app conventions. Buttons are evenly spaced with icon + short label.
- On desktop (above the breakpoint), the tab bar sits at the **top** below the header, as a horizontal nav row. Both placements work; the bottom-bar is critical for thumb reach on phones.
- Breakpoint: reuse the existing `720px` mobile breakpoint.

### Desktop behavior

- The tab bar is a horizontal row below the page header, above the tab panels.
- Tab panels use the same grid layout as today (`repeat(auto-fit, minmax(320px, 1fr))`) within each panel.
- On wide screens, the Shop tab shows Pantry and Ingredient Matches side-by-side (half-width panels) with Shopping List spanning full width above — matching the current layout but scoped to fewer panels.

## Accessibility

- Tab buttons use `role="tab"` with `aria-selected`.
- Tab panels use `role="tabpanel"` with `aria-labelledby` pointing to their tab button's `id`.
- Arrow-key navigation within the tab bar (left/right moves focus between tabs, Enter/Space activates). This follows the [WAI-ARIA Tabs pattern](https://www.w3.org/WAI/ARIA/apg/patterns/tabs/).
- The tab bar container uses `role="tablist"`.
- Focus management: when switching tabs, focus stays on the tab button (not moved into the panel). The user can then Tab into the panel content.
- Status banner remains `role="status"` with `aria-live="polite"` (already addressed in cleanup backlog item #1).

## Cross-Tab Interactions

| Trigger | Source tab | Target tab | Behavior |
|---|---|---|---|
| "Generate shopping list from plan" | Plan | Shop | Generate via API, switch to Shop tab, render result. |
| "Copy plan & shopping list" | Plan | (stays on Plan) | Generate via API, copy combined text, update Shop state in background. No tab switch — the output is the clipboard. |
| "Add to list" on meal card | Meals | (stays on Meals) | Toggle the meal in `state.shoppingMealIds`. The Shop tab will reflect this when the user switches to it. Show status banner confirmation. |
| "Edit" on a meal from picker/plan context | Plan | Meals | Switch to Meals tab, scroll to and populate the meal editor form. |
| Shopping list "Generate" → recipe links | Shop | (stays on Shop) | No cross-tab needed. |

## Acceptance Criteria

### Structure

1. Three tabs are visible: Plan, Shop, Meals.
2. Only one tab's content is visible at a time.
3. Switching tabs hides/shows the correct panels instantly.
4. The Plan tab is the default active tab on first load.
5. The active tab persists across page refreshes (via `sessionStorage`).

### Mobile

6. The tab bar is fixed at the bottom of the screen on viewports ≤720px.
7. Tab buttons are thumb-reachable and evenly spaced.
8. All panel content within each tab scrolls normally.
9. The status banner remains visible above the tab content.

### Desktop

10. The tab bar is positioned below the page header on viewports >720px.
11. Within the Shop tab, Pantry and Ingredient Matches appear side-by-side on wide screens.

### Cross-Tab

12. "Generate shopping list from plan" switches to the Shop tab and shows the result.
13. "Copy plan & shopping list" stays on Plan tab (output is clipboard, not a panel).
14. "Add to list" on a meal card stays on Meals tab and shows a status confirmation.
15. "Edit" on a meal (from any context) switches to the Meals tab and populates the form.

### Accessibility

16. Tab buttons have `role="tab"` with correct `aria-selected`.
17. Tab panels have `role="tabpanel"` with `aria-labelledby`.
18. Arrow keys navigate between tab buttons.
19. Screen readers can identify the current tab and navigate to any tab by landmark.

### Non-Regression

20. All existing features (plan, autofill, random, serve, pantry, matches, shopping list, copy, share, history, meal CRUD) continue to work within their new tab locations.
21. No backend changes required.
22. Existing tests pass without modification (they test backend APIs, not frontend layout).

## Risks and Assumptions

| Risk / Assumption | Impact | Mitigation |
|---|---|---|
| Moving panels into tab containers may break existing `elements.*` references if DOM structure changes. | Medium — querySelector selectors are by ID, not by parent. | All `getElementById` / `querySelector("#...")` calls work regardless of tab container nesting. Verify manually. |
| Cross-tab `scrollIntoView` calls (e.g., `sendPlanToShoppingList` scrolls to `#shopping-list-panel`) won't work if the target is on a hidden tab. | Low — these are identified above and must switch tabs first. | The implementation explicitly switches tab before scrolling. |
| `display: none` on inactive tabs means forms retain their state (good) but may cause layout thrashing on tab switch (unlikely). | Very low — the DOM is small. | No performance concern for a family-scale app. |
| The tab bar at bottom on mobile may conflict with browser chrome (e.g., Safari's bottom bar). | Low — standard pattern used by most mobile web apps. | Use `padding-bottom` on the page body to account for tab bar height. Test on iOS Safari. |
| Users may briefly not find a feature after the restructuring. | Low — three tabs with clear labels. | The tab names map to user intent ("plan dinner," "shop," "manage meals"). First-time confusion resolves within one session. |
