# V1.5 Weekly Share Pack — Combined Plan + Shopping List Copy

## Problem Statement

The app can now copy the weekly dinner plan (v1.4) and the shopping list (v1.1) separately, but the most common sharing scenario is "here's what we're eating this week and here's what we need to buy" — a single message to the family. Today the user has to: copy the plan, paste it, switch back, generate the shopping list, copy it, paste it below. That is four clipboard round-trips for one share.

A one-button "Copy plan & shopping list" action produces a single combined text block ready to paste into a note, text message, or Google Doc.

## Current Repo Context

Inspected files relevant to this slice:

- **`public/app.js` → `formatWeeklyPlanText(plan)`** (line ~1159) — pure function, takes `state.currentPlan`, returns the plan as plain text. Already used by `copyWeeklyPlan()` and `copyArchivedPlan()`.
- **`public/app.js` → `state.shoppingListResult.copyText`** — the shopping list `copyText` is a plain-text string generated server-side by `formatShoppingListPlainText()` in `src/modules/shopping-list/shopping-list.service.js`. It is available in client state after any call to `POST /api/shopping-list/generate`.
- **`public/app.js` → `sendPlanToShoppingList()`** (line ~1010) — the existing "Generate shopping list from plan" button handler. Collects non-null meal IDs from plan slots, reads the shopping-list form settings (`useSavedPantry`, `includeOptional`, `ingredientNames`), calls `POST /api/shopping-list/generate`, stores the result in `state.shoppingListResult`, renders it, and scrolls to the shopping-list panel.
- **`public/app.js` → `copyShoppingList()` / `copyWeeklyPlan()`** — both use `navigator.clipboard.writeText` with `showStatus()` feedback. Same pattern to reuse.
- **`public/app.js` → `renderWeeklyPlan()`** — the plan footer already contains "Generate shopping list from plan" and "Copy plan" buttons (when `plannedMealCount > 0`). The new button goes here.
- **`public/app.js` → `handleWeeklyPlanActions()`** (line ~1070) — dispatches plan-panel button clicks by `data-action`.

## Decision: Frontend-Only with One API Call

**No new backend endpoint.** The combined text is a concatenation of two pieces that already exist:

1. **Plan text** — generated client-side by `formatWeeklyPlanText(state.currentPlan)`. Pure function, no API call, already implemented.
2. **Shopping list text** — `state.shoppingListResult.copyText`, which requires `POST /api/shopping-list/generate`. This API call is already implemented in `sendPlanToShoppingList()`.

The "share pack" button needs to:
1. Generate the shopping list from the plan (if not already generated for the current plan state), using the same API call and form settings as the existing flow.
2. Concatenate the plan text and shopping list `copyText` with a separator.
3. Copy the combined text to the clipboard.

**Why not a new backend endpoint?** The plan text is trivial formatting of client state. The shopping list `copyText` is already returned by the existing endpoint. Combining them server-side would require a new endpoint that duplicates logic from two unrelated modules — more code, more testing surface, and no benefit. The frontend has both pieces.

## User Flow

1. User has an active weekly plan with at least one meal assigned.
2. User clicks **"Copy plan & shopping list"** in the plan panel footer.
3. The app generates the shopping list for the plan's meals (calling the existing API with the current shopping-list form settings: use-saved-pantry, include-optional, extra ingredients).
4. The plan text and shopping-list text are combined into one block.
5. The combined text is copied to the clipboard.
6. The status banner shows "Plan and shopping list copied to clipboard."

If the shopping list was already generated for the same plan meals (via a prior "Generate shopping list from plan" click), **the button still re-generates** to ensure the list reflects the latest pantry/form state. This keeps the behavior simple and predictable — no stale-cache guessing.

### Error cases

- If the plan has no meals assigned, the button is not shown (same guard as the existing plan-footer buttons).
- If the shopping-list API call fails, the error is shown in the status banner and nothing is copied.
- If the clipboard write fails, the fallback error is shown.

## Combined Output Format

```
Dinner plan — week of Apr 6, 2026

Monday    Apr 6   Spaghetti Tacos ✓
Tuesday   Apr 7   Black Bean Quesadillas — use leftover rice
Wednesday Apr 8   (no plan)
Thursday  Apr 9   Chicken Stir-Fry
Friday    Apr 10  Grilled Salmon
Saturday  Apr 11  (no plan)
Sunday    Apr 12  Pizza Night ★

---

Shopping list for: Spaghetti Tacos, Black Bean Quesadillas, Chicken Stir-Fry, Grilled Salmon, Pizza Night

Need to buy:
- Taco shells — Spaghetti Tacos (8 shells)
- Salmon fillets — Grilled Salmon (2 fillets)
- Pizza dough — Pizza Night (1 ball)

Optional:
- Salsa — Black Bean Quesadillas (for serving)
```

### Format rules

| Element | Representation | Rationale |
|---|---|---|
| **Plan section** | Verbatim output of `formatWeeklyPlanText(plan)` | Reuses the existing formatter with no modifications. |
| **Separator** | `---` on its own line, with a blank line above and below | Visually distinct in plain text, Markdown-friendly, no ambiguity. |
| **Shopping list section** | Verbatim `copyText` from the shopping-list API response | Reuses the backend-generated format. No modifications. |
| **Trailing newline** | One at the end | Clean paste behavior. |

### Why verbatim concatenation

- Both individual texts are already designed to be self-contained and paste-friendly.
- Reformatting either section for the combined view would create a maintenance burden — any future change to either formatter would need to be mirrored.
- The `---` separator is universally understood and renders well in plain text, Markdown, and Google Docs.

### What appears from each section

**From the plan text** (via `formatWeeklyPlanText`):
- Header with week date
- All 7 days with meal names / `(no plan)`
- `✓` for served, `★` for favorites, `(archived)` for archived meals
- Slot notes appended with `—`

**From the shopping list `copyText`** (via `formatShoppingListPlainText`):
- "Shopping list for:" header with meal names
- "Need to buy:" section with ingredient names and quantity hints
- "Optional:" section (only when `includeOptional` was checked and optional items exist)
- Does **not** include "Already on hand" — the existing `copyText` format intentionally omits on-hand items for a clean buy-list

### Shopping-list form settings

The share-pack button reads the same form settings as the existing shopping-list flow:
- **Use saved pantry** — checkbox, checked by default
- **Include optional ingredients** — checkbox
- **Extra on-hand ingredients** — textarea, one per line

These are read from the shopping-list form DOM at generation time, matching the behavior of both `generateShoppingList()` and `sendPlanToShoppingList()`. The user can adjust these settings in the Shopping List panel before clicking the share-pack button, and the generated list will reflect them.

## In Scope

- A `copyPlanAndShoppingList()` async function in `public/app.js`.
- A "Copy plan & shopping list" button in the plan panel footer.
- Generating the shopping list on demand via the existing API.
- Concatenating plan text + separator + shopping `copyText`.
- Clipboard write with status-banner feedback.

## Out of Scope

- Modifying either `formatWeeklyPlanText` or `formatShoppingListPlainText`.
- A combined preview panel (the individual plan and shopping list panels are visible on-screen).
- Customizable separator or output format.
- Persisting the combined text.
- Applying this to archived plans (requires generating a shopping list from an archived plan's meals, which the existing API supports but the UI flow doesn't — natural follow-up but out of scope).

## Interaction with Existing Features

### Existing "Copy plan" button

Stays as-is. Copies only the plan text. The share-pack button is additive — users who just want the dinner schedule can still use "Copy plan."

### Existing "Copy to clipboard" on the shopping list

Stays as-is. Copies only the shopping list text. Independent of the share-pack flow.

### Existing "Generate shopping list from plan" button

Stays as-is. The share-pack button generates the shopping list internally using the same logic, but does not replace or modify the shopping-list panel state. **However**, to avoid hidden state divergence, the share-pack handler **should** also update `state.shoppingListResult` and render the shopping-list panel — this way the user sees the same list they just copied if they scroll down. This matches what `sendPlanToShoppingList()` already does.

### Shopping-list form settings

The share-pack button reads from `elements.shoppingListForm` at click time — same as `sendPlanToShoppingList()`. If the user changes the pantry checkbox or adds extra on-hand ingredients before clicking, the share pack reflects those settings.

### Slot notes

Included in the plan section of the combined output, same as the standalone plan copy.

### Served status

Included as `✓` in the plan section. The shopping list is generated from all non-null meal IDs regardless of served status — a served meal still needs its ingredients bought if the family hasn't shopped yet.

## Acceptance Criteria

1. A "Copy plan & shopping list" button appears in the plan footer when at least one meal is planned.
2. Clicking the button generates a shopping list from the plan's meals using the current shopping-list form settings.
3. The combined text is copied to the clipboard.
4. The combined text contains the plan section, a `---` separator, and the shopping list section.
5. The plan section matches the output of `formatWeeklyPlanText`.
6. The shopping list section matches the `copyText` from the API response.
7. The status banner shows "Plan and shopping list copied to clipboard" on success.
8. If the API call fails, the error is shown in the status banner and nothing is copied.
9. If the clipboard write fails, a fallback error is shown.
10. After the operation, `state.shoppingListResult` is updated and the shopping-list panel renders the result (same side effect as "Generate shopping list from plan").
11. The existing "Copy plan" and "Copy to clipboard" (shopping list) buttons continue to work independently.

## Risks and Assumptions

| Risk / Assumption | Impact | Mitigation |
|---|---|---|
| Re-generating the shopping list on every share-pack click adds a small API round-trip. | Low — the API call is fast (in-memory SQLite, no external deps). | Acceptable. Ensures the list always reflects current pantry state. |
| The combined output may be long for 7 meals with many ingredients. | Low — family meal plans are typically 5–7 distinct meals with moderate ingredient lists. | Plain text keeps it scannable. The format is already proven by the individual copy features. |
| Reading form settings from the shopping-list panel means the user must have those settings visible/configured. | Low — defaults are sensible (pantry on, optional off, no extras). | Same assumption the existing "Generate shopping list from plan" already makes. |
| `navigator.clipboard.writeText` length limits. | Very low — clipboard APIs handle multi-KB text without issue. | No practical limit for a 7-meal plan + shopping list. |

## Implementation Tasks

### Frontend-Only Work Package

**Scope:** `public/app.js`

**No backend changes. No migration. No new endpoints.**

**Work:**

1. **Add `copyPlanAndShoppingList()` async function:**
   - Guards on `state.currentPlan` and at least one planned meal.
   - Collects plan meal IDs (same logic as `sendPlanToShoppingList`).
   - Reads shopping-list form settings from DOM (same logic as `sendPlanToShoppingList`).
   - Calls `POST /api/shopping-list/generate` via `apiFetch` (same call as the existing flow).
   - Updates `state.shoppingListResult` and renders the shopping-list panel (mirrors `sendPlanToShoppingList` side effects).
   - Builds the combined text: `formatWeeklyPlanText(state.currentPlan)` + `\n---\n\n` + `result.copyText` + `\n`.
   - Calls `navigator.clipboard.writeText(combinedText)`.
   - Shows `showStatus("Plan and shopping list copied to clipboard")`.
   - Catches API errors and clipboard errors with `showStatus(..., "error")`.

   Implementation sketch:
   ```js
   async function copyPlanAndShoppingList() {
     if (!state.currentPlan) return;

     const slots = state.currentPlan.slots.filter(s => s.meal !== null);
     if (slots.length === 0) {
       showStatus("No meals planned yet", "error");
       return;
     }

     const uniqueIds = [...new Set(slots.map(s => s.meal.id))];

     // Read shopping-list form settings
     const formData = new FormData(elements.shoppingListForm);
     const ingredientNames = String(formData.get("ingredientNames") || "")
       .split("\n").map(v => v.trim()).filter(Boolean);

     const payload = {
       mealIds: uniqueIds,
       useSavedPantry: formData.get("useSavedPantry") === "on",
       includeOptional: formData.get("includeOptional") === "on",
     };
     if (ingredientNames.length > 0) payload.ingredientNames = ingredientNames;

     try {
       const result = await apiFetch("/api/shopping-list/generate", {
         method: "POST",
         body: JSON.stringify(payload),
       });

       // Update shopping-list panel state (same side effect as sendPlanToShoppingList)
       state.shoppingMealIds = new Set(uniqueIds);
       state.shoppingMealOverrides = new Map();
       slots.forEach(s => state.shoppingMealOverrides.set(s.meal.id, s.meal));
       state.shoppingListResult = result;
       renderShoppingSelection();
       renderShoppingListResult(result);
       renderMeals();

       // Build combined text
       const planText = formatWeeklyPlanText(state.currentPlan);
       const combinedText = planText + "\n---\n\n" + result.copyText + "\n";

       await navigator.clipboard.writeText(combinedText);
       showStatus("Plan and shopping list copied to clipboard");
     } catch (error) {
       showStatus(error.message, "error");
     }
   }
   ```

   Note: the shopping-list generation logic (collecting IDs, reading form settings, calling the API, updating state) is duplicated from `sendPlanToShoppingList`. If preferred, the implementer may extract a shared helper like `generateShoppingListForPlan()` that both functions call, to reduce duplication. This is a minor code-quality improvement, not a functional requirement.

2. **Add "Copy plan & shopping list" button to `renderWeeklyPlan()`:**
   - In the `plan-footer` div, add a button alongside the existing two:
     ```html
     <button type="button" data-action="plan-share-pack">Copy plan & shopping list</button>
     ```
   - This is the primary sharing action, so it should be visually prominent (no `class="secondary"`). The existing "Generate shopping list from plan" becomes secondary, and "Copy plan" stays secondary.
   - The footer only renders when `plannedMealCount > 0` — no change needed to the guard.

3. **Wire the action in `handleWeeklyPlanActions()`:**
   - Add a case for `action === "plan-share-pack"` that calls `copyPlanAndShoppingList()`.

4. **(Optional) Extract shared shopping-list generation helper:**

Both callers then use it and add their own behavior (scroll vs. copy). This is a refactor opportunity, not a blocker.
