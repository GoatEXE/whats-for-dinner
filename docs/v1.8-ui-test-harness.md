## Problem Statement

The app has 42 backend tests (Vitest + Supertest) covering API behavior, but zero frontend verification. Workers making layout and UX changes — like the recent tabbed restructuring (v1.7), mobile plan-slot adjustments, and cross-tab navigation wiring — have no automated way to check that the UI still works. Every change requires manual browser testing across viewports, which is slow, error-prone, and often skipped.

The user wants a lightweight test harness so workers can make UI adjustments with more confidence. The goal is not comprehensive E2E coverage — it is a small, fast smoke-test suite that catches obvious breakage across the key screens and viewports.

## Recommendation: Playwright

**Use Playwright** with a minimal smoke-test suite.

### Why Playwright

- **First-party viewport/device emulation.** Playwright supports `--project` configs for desktop and mobile viewports out of the box — critical for validating the tabbed layout at different sizes.
- **Built-in screenshot comparison.** `expect(page).toHaveScreenshot()` generates baseline PNGs and diffs on subsequent runs. Workers can see exactly what changed in the UI without manually opening a browser.
- **Headless by default.** Runs in CI and on dev machines without a display server. Fast.
- **No dependency on the existing Vitest config.** Playwright has its own config file and test runner (`npx playwright test`). This avoids disrupting the working Vitest setup or mixing browser and node test environments.
- **Lightweight install.** `npm install -D @playwright/test` + `npx playwright install chromium` installs just one browser engine (~130MB). No need for the full browser suite.

### Why not other options

- **Vitest browser mode** (`@vitest/browser`) would require reconfiguring the existing `vitest.config.js` to support both `environment: "node"` (current tests) and a browser environment (new tests). This is more disruptive than adding a separate Playwright config.
- **Puppeteer** requires manual viewport handling, no built-in screenshot diffing, and more boilerplate for the same result.
- **Cypress** is heavier, has a different testing model (time-travel debugging), and is overkill for a smoke-test harness.

## What To Test First

The harness should cover the **highest-risk UI surfaces** — the areas most likely to break during layout changes. These are not exhaustive feature tests (the backend Supertest suite covers API correctness); they verify that the UI renders correctly and the key interactions work.

### Tier 1: Structural smoke tests (must-have)

| Test | What it verifies | Why it's high-risk |
|---|---|---|
| **Page loads and tabs render** | Title visible, 3 tab buttons present and labeled, Plan tab active by default | Any HTML/CSS restructuring could break the shell |
| **Tab switching** | Click each tab → correct panel visible, others hidden | The `switchTab` function and `.hidden` toggling are central to all UI |
| **Mobile tab bar position** | At 375px width, tab bar is at the bottom of the viewport | Mobile bottom-bar is critical UX and easy to break with CSS changes |
| **Meals tab: meal cards render** | With seeded data, meal cards appear with name, ingredients, action buttons | The most content-rich panel; layout issues surface here |
| **Plan tab: weekly plan renders** | After creating a plan via API, 7 day slots are visible with labels | The most complex panel; slot grid is fragile on mobile |
| **Shop tab: pantry list renders** | Seeded pantry items are visible | Verifies the Shop tab's content loads |

### Tier 2: Interaction smoke tests (should-have)

| Test | What it verifies |
|---|---|
| **Create a meal via the form** | Fill the meal form, submit, verify the new card appears in the meals list |
| **Quick picker returns a result** | Click "Pick dinner" → result card appears with a meal name |
| **Generate shopping list from plan** | Create a plan, assign meals, click "Generate shopping list" → switches to Shop tab, shows results |
| **Copy plan to clipboard** | Click "Copy plan" → status banner shows success |

### Tier 3: Visual regression screenshots (nice-to-have)

| Screenshot | Viewports |
|---|---|
| Plan tab (empty plan state) | Desktop 1280×800, Mobile 375×812 |
| Plan tab (populated plan) | Desktop, Mobile |
| Meals tab (with meal cards) | Desktop, Mobile |
| Shop tab (with pantry + empty shopping list) | Desktop, Mobile |

Screenshots use `toHaveScreenshot()` with a threshold to absorb font-rendering differences across platforms. Baseline PNGs are committed to the repo so diffs are visible in PRs.

## Viewport Strategy

Two viewports cover the important breakpoints:

| Name | Width × Height | Rationale |
|---|---|---|
| **Desktop** | 1280 × 800 | Standard laptop. Tests the top-bar tab layout, side-by-side panel grid, plan-slot 3-column grid. |
| **Mobile** | 375 × 812 | iPhone SE / small Android. Tests bottom-bar tabs, stacked panels, single-column plan slots. Matches the `≤720px` CSS breakpoint. |

These are configured as Playwright projects in `playwright.config.js`, so every test runs at both viewports automatically.

## Test Infrastructure Design

### File structure

```
e2e/
  fixtures/
    test-server.js          # Starts app with temp DB + seed data, returns { baseURL, cleanup }
  tests/
    tabs.spec.js            # Tab structure and switching
    meals.spec.js           # Meal cards, create meal
    plan.spec.js            # Weekly plan rendering, picker
    shop.spec.js            # Pantry, shopping list
  playwright.config.js      # Playwright config (viewports, baseURL, screenshot dir)
```

### Test server fixture

The E2E tests need a running HTTP server (unlike Supertest which uses the app directly). A shared Playwright fixture will:

1. Create a temp SQLite DB (same pattern as `test/helpers/test-app.js`).
2. Run migrations + seed data.
3. Start the Express app on a random port.
4. Provide `baseURL` to all tests.
5. Clean up the temp DB and stop the server after all tests.

This is a Playwright `globalSetup` or a test fixture — the fixture approach is simpler and avoids global state.

```js
// e2e/fixtures/test-server.js
const { createTestContext } = require("../../test/helpers/test-app");
const { seedDatabase } = require("../../src/db/seed");

async function startTestServer() {
  const context = createTestContext();
  seedDatabase(context.db);

  const server = await new Promise((resolve) => {
    const srv = context.app.listen(0, () => resolve(srv));
  });

  const port = server.address().port;

  return {
    baseURL: `http://localhost:${port}`,
    async cleanup() {
      await new Promise((resolve) => server.close(resolve));
      context.cleanup();
    },
  };
}

module.exports = { startTestServer };
```

### Playwright config

```js
// e2e/playwright.config.js
const { defineConfig, devices } = require("@playwright/test");

module.exports = defineConfig({
  testDir: "./tests",
  timeout: 15000,
  retries: 0,
  use: {
    headless: true,
    screenshot: "only-on-failure",
  },
  projects: [
    {
      name: "desktop",
      use: { viewport: { width: 1280, height: 800 } },
    },
    {
      name: "mobile",
      use: { viewport: { width: 375, height: 812 } },
    },
  ],
});
```

### Test commands

```json
{
  "scripts": {
    "test": "vitest run",
    "test:ui": "playwright test --config e2e/playwright.config.js",
    "test:ui:update": "playwright test --config e2e/playwright.config.js --update-snapshots"
  }
}
```

- `npm test` — existing backend tests (unchanged).
- `npm run test:ui` — Playwright smoke tests at both viewports.
- `npm run test:ui:update` — regenerate screenshot baselines.

### Screenshot storage

Screenshots are stored in `e2e/tests/__screenshots__/` (Playwright's default with `toHaveScreenshot()`). Baseline PNGs should be committed to the repo. `.gitignore` should exclude `e2e/test-results/` (Playwright's failure artifact directory).

## Risks and Assumptions

| Risk / Assumption | Impact | Mitigation |
|---|---|---|
| Playwright install adds ~130MB for Chromium. | Low — dev dependency only, not shipped in Docker. | `@playwright/test` is in `devDependencies`. Docker image uses `--omit=dev`. |
| Screenshot baselines may differ across OS/platform due to font rendering. | Medium — a macOS dev and a Windows dev may generate different baselines. | Use a reasonable `maxDiffPixelRatio` threshold (e.g., 0.01). For strict consistency, run screenshot updates in a consistent environment (or defer screenshots to CI later). |
| Starting a real HTTP server per test file adds startup time. | Low — the Express + SQLite app starts in <100ms. | Use a per-worker fixture that starts once and shares across tests in the same file. |
| Playwright requires Node 18+. | None — the project already requires Node 22. |
| Tests may be flaky if `loadData()` or `runPantryMatch()` is slow. | Very low — in-memory SQLite with 3 meals. | Add `waitForSelector` on key elements before assertions. |
| The existing `createTestContext()` helper is CommonJS. Playwright tests default to ESM. | Low — Playwright supports CommonJS via config. | Use `require()` in test files, or set `testDir` config accordingly. The existing test helper is already CommonJS. |
