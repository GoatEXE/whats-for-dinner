# V1.6 Autofill Empty Plan Slots

## Problem Statement

The weekly planner requires the user to fill each day individually — either by picking a meal from the dropdown or clicking "Random" per slot. For a family starting with a mostly-empty plan, that is seven separate interactions to get a full week. The existing "Use as starting point" (v1.3) helps when reusing a past week, but when starting fresh or filling gaps in a partial plan, there is no batch action.

A one-click "Fill empty days" button would randomly assign meals to all unoccupied slots at once, respecting the same duplicate-avoidance and recently-served filters as the per-slot random fill. The user can then swap out any picks they don't like. This reduces Sunday planning from 7 decisions to 0–2 tweaks.

## Current Repo Context

Inspected files relevant to this slice:

- **`src/modules/weekly-plans/weekly-plans.service.js` → `fillSlotRandom(day, filters)`** — the per-slot random fill. Gets the plan, reads planned meal IDs via `weeklyPlansRepo.listPlannedMealIds(plan.id)`, builds an `excludeMealIds` list (planned meals minus the current slot's meal if it's unique to that slot), calls `suggestionsService.pickRandomMeal(filters)`, then writes the slot via `weeklyPlansRepo.updateSlot`. Throws `HttpError(404)` when no candidates remain.
- **`src/modules/weekly-plans/weekly-plans.repo.js` → `updateSlot(planId, day, mealId, notes)`** — upserts a slot and bumps `weekly_plans.updated_at` in a transaction. Returns the full rebuilt plan.
- **`src/modules/weekly-plans/weekly-plans.repo.js` → `listPlannedMealIds(planId)`** — returns deduplicated non-null meal IDs currently assigned to slots.
- **`src/modules/weekly-plans/weekly-plans.schemas.js` → `randomSlotSchema`** — validates `{ favoritesOnly, fullMatchOnly, excludeServedWithinDays, excludePlannedMeals }` with sensible defaults.
- **`public/app.js` → `fillPlanSlotRandom(day)`** (line ~977) — frontend handler for the per-slot "Random" button. Calls `POST /api/weekly-plans/current/slots/:day/random` with `{}` (all defaults).
- **`public/app.js` → `renderWeeklyPlan()`** (line ~830) — renders the plan panel. The `plan-footer` div (line ~927) conditionally shows buttons when `plannedMealCount > 0`. The "Fill empty" button needs to appear even when `plannedMealCount === 0`.

## Decision: Backend Batch Endpoint

**Add `POST /api/weekly-plans/current/autofill`.** Rationale:

1. **Sequential per-slot calls from the frontend would make up to 7 HTTP round-trips**, each re-reading the full plan (slot joins + served-status lookup), re-querying candidates, and returning the full plan object. A backend batch avoids this overhead.
2. **Deduplication must be maintained across fills.** Each new assignment must be excluded from subsequent picks within the same batch. Doing this client-side would require waiting for each response before firing the next call — effectively serial anyway, but with HTTP overhead per step.
3. **The `pickRandomMeal` 404 (no candidates) needs graceful partial handling.** If the user has 3 empty slots but only 2 candidate meals remain, the batch should fill 2 and leave 1 empty — not fail entirely. This is easier to handle in a single server-side loop than across 7 independent frontend calls where the 3rd call returns 404 and the frontend must decide what to do.
4. **The implementation is thin.** It is a loop over existing service functions with an accumulating exclude list, wrapped in a single transaction.

## User Flow

1. User has an active weekly plan with some empty slots.
2. User clicks **"Fill empty days"** in the plan panel.
3. The app calls the autofill endpoint, which randomly assigns meals to all empty slots.
4. The plan panel re-renders showing the filled slots.
5. The status banner shows "Filled N empty days" (or "No empty days to fill" if already full).
6. User reviews and tweaks individual slots as needed — swap, clear, or re-random any slot.

## What Gets Filled vs. Preserved

| Slot state | Autofill behavior |
|---|---|
| Empty (no meal) | Filled with a random pick. |
| Has a meal (not served) | **Preserved.** Not touched. |
| Has a meal (served) | **Preserved.** Not touched. |
| Has notes but no meal | Notes are **preserved**. The slot is filled with a random meal and the existing notes carry through. |

This is the key design decision: autofill only targets slots where `meal_id IS NULL`. It never overwrites an existing assignment. Notes on empty slots (unlikely but possible) survive.

## API Design

### `POST /api/weekly-plans/current/autofill`

Fill all empty slots in the current plan with random meals.

**Request body:**
```json
{
  "favoritesOnly": false,
  "fullMatchOnly": false,
  "excludeServedWithinDays": 7
}
```

All fields are optional with sensible defaults (matching `randomSlotSchema`). `excludePlannedMeals` is always `true` internally — there is no reason to allow duplicate meals in a batch fill.

**Response:** `200` with the `{ data: ... }` envelope containing the updated full plan object (same shape as all other plan-mutating endpoints).

**Response also includes a summary header or body field:**
```json
{
  "data": {
    "id": 1,
    "weekStart": "2026-04-06",
    "isArchived": false,
    "slots": [ ... ],
    "autofillResult": {
      "filledCount": 4,
      "skippedCount": 1,
      "emptyBeforeCount": 5,
      "noMoreCandidates": true
    }
  }
}
```

- `filledCount` — how many empty slots were successfully filled.
- `skippedCount` — how many empty slots could not be filled because candidates ran out.
- `emptyBeforeCount` — how many slots were empty before the operation.
- `noMoreCandidates` — `true` if the loop stopped early because `pickRandomMeal` exhausted candidates. `false` if all empty slots were filled.

This metadata lets the frontend show a precise status message.

**Behavior when no empty slots exist:** returns `200` with the unmodified plan and `autofillResult.emptyBeforeCount === 0`. Not an error — the plan is already full.

**Behavior when candidates run out mid-fill:** fills as many slots as possible, leaves the rest empty, returns success with the partial result. Not an error — the user can see which slots were filled and handle the rest manually.

### Why not reuse `randomSlotSchema`?

The autofill schema is a subset of `randomSlotSchema` — it shares `favoritesOnly`, `fullMatchOnly`, and `excludeServedWithinDays`, but omits `excludePlannedMeals` (always true). A new `autofillSchema` is cleaner than special-casing the existing schema.

## Autofill Algorithm

```
1. Load the current active plan.
2. Identify empty slots (meal_id IS NULL), sorted by day (0–6).
3. Build the initial exclude set: all non-null meal IDs already in the plan.
4. For each empty slot, in day order:
   a. Call pickRandomMeal({ ...filters, excludeMealIds: [...excludeSet] }).
   b. If a meal is returned:
      - Write the slot (preserving existing notes).
      - Add the new meal ID to the exclude set.
      - Increment filledCount.
   c. If pickRandomMeal throws 404 (no candidates):
      - Stop the loop. Remaining empty slots cannot be filled.
      - Set noMoreCandidates = true.
      - Increment skippedCount for each remaining empty slot.
5. Rebuild and return the updated plan.
```

### Why fill in day order?

Deterministic and predictable. The user sees Monday filled first, then Tuesday, etc. If candidates run out, the gaps are at the end of the week, which feels natural (the family can figure out the weekend later).

### Duplicate avoidance

The exclude set grows as meals are assigned. Combined with `excludePlannedMeals` behavior, no meal appears twice in the plan unless the user deliberately assigns duplicates via the per-slot manual picker.

### Interaction with `excludeServedWithinDays`

The `excludeServedWithinDays` filter is applied globally by `pickRandomMeal` — meals served within that window are excluded from candidates before the random pick. This interacts correctly with the exclude set: a meal that was served 3 days ago and is excluded by `excludeServedWithinDays=7` will not be picked for any slot, regardless of the growing exclude set.

## Interaction with Existing Features

### Per-slot "Random" button

Stays as-is. The user can still random-fill individual slots. Autofill is a convenience for batch filling, not a replacement.

### Plan notes

Autofill preserves existing notes on empty slots. The `updateSlot` repo function already accepts notes — autofill passes through the slot's current notes (typically null for empty slots, but could be non-null if the user typed a note before assigning a meal).

### Favorites

If `favoritesOnly: true` is passed, autofill only picks from favorited meals. This matches the per-slot random behavior.

### Pantry / full-match-only

If `fullMatchOnly: true` is passed, autofill only picks meals whose required ingredients are all in the pantry. This matches the per-slot random behavior — it delegates to the same `pickRandomMeal` function with the same filter.

### History / recently-served

`excludeServedWithinDays` works the same as in per-slot random. Default is `0` (no exclusion) matching the existing `randomSlotSchema` default.

### Plan reuse (v1.3)

No conflict. A reused plan may have some empty slots (if the source had gaps). Autofill fills those gaps.

### Copy / share pack (v1.4, v1.5)

No conflict. Autofill mutates the plan state. Copy/share reads from the updated state.

### Shopping list

No direct interaction. After autofill, the user can generate a shopping list as usual.

## Acceptance Criteria

### Backend

1. `POST /api/weekly-plans/current/autofill` fills all empty slots with random meals and returns `200` with the updated plan.
2. Already-occupied slots are not modified.
3. Each filled slot gets a unique meal (no duplicates within the plan, assuming sufficient candidates).
4. If candidates run out before all empty slots are filled, the response includes the partial result with `noMoreCandidates: true` and `skippedCount > 0`.
5. If no empty slots exist, the response returns the unmodified plan with `filledCount: 0`.
6. `favoritesOnly`, `fullMatchOnly`, and `excludeServedWithinDays` filters are respected.
7. Slot notes are preserved (not cleared) during autofill.
8. The plan's `updated_at` is bumped if any slots were filled.
9. Returns `404` if no active plan exists.
10. The `autofillResult` summary in the response accurately reflects the operation.

### Frontend

1. A "Fill empty days" button appears in the plan panel when at least one empty slot exists.
2. Clicking the button calls the autofill endpoint.
3. The plan panel re-renders with the filled slots.
4. The status banner shows "Filled N empty days" (or the appropriate message for partial fills / no empties).
5. If candidates ran out, the message indicates some slots could not be filled.
6. The button is hidden or disabled when all 7 slots are occupied.

## Risks and Assumptions

| Risk / Assumption | Impact | Mitigation |
|---|---|---|
| A household with very few meals (e.g. 3) cannot fill a 7-day plan without duplicates. | Low — partial fill is the designed behavior. | `autofillResult` clearly communicates how many were filled vs skipped. Status message tells the user. |
| `pickRandomMeal` is called up to 7 times in a single request, each querying candidate meals. | Low — all queries are in-memory SQLite on a local app. | Acceptable. The candidate-meal query is already fast for single-household data. |
| Autofill always fills Monday→Sunday. Some users might prefer weekend-first. | Very low — this is a family dinner tool, not an enterprise scheduler. | Day-order is deterministic and predictable. Custom fill order is not worth the complexity. |
| Users might want to exclude specific meals from autofill (not just recently-served). | Low — the user can clear and re-fill individual slots after autofill. | The clear + per-slot random flow already handles this. A per-meal exclude list is over-engineering for v1.6. |

## Implementation Tasks

### Backend Work Package 1: Service and Route

**Scope:** `src/modules/weekly-plans/weekly-plans.service.js`, `src/modules/weekly-plans/weekly-plans.schemas.js`, `src/modules/weekly-plans/weekly-plans.routes.js`

**Work:**

**Schema (`weekly-plans.schemas.js`):**
- Add `autofillSchema`:
  ```js
  const autofillSchema = z
    .object({
      favoritesOnly: booleanish.optional().default(false),
      fullMatchOnly: booleanish.optional().default(false),
      excludeServedWithinDays: z.coerce
        .number()
        .int()
        .min(0)
        .max(365)
        .optional()
        .default(0),
    })
    .default({});
  ```
  This is a subset of `randomSlotSchema` without `excludePlannedMeals`.

**Service (`weekly-plans.service.js`):**
- Add `autofillEmptySlots(filters)` function:
  - Load current plan via `getCurrentPlan()`.
  - Identify empty slots: `plan.slots.filter(s => s.meal === null)`, sorted by `day`.
  - If no empty slots, return `{ plan, autofillResult: { filledCount: 0, skippedCount: 0, emptyBeforeCount: 0, noMoreCandidates: false } }`.
  - Build initial exclude set from `weeklyPlansRepo.listPlannedMealIds(plan.id)`.
  - Loop over empty slots:
    - Try `suggestionsService.pickRandomMeal({ ...filters, excludeMealIds: [...excludeSet] })`.
    - On success: call `weeklyPlansRepo.updateSlot(plan.id, slot.day, randomResult.meal.id, slot.notes)` to write the slot (preserving notes). Add meal ID to exclude set. Increment `filledCount`.
    - On `HttpError(404)` (no candidates): set `noMoreCandidates = true`, count remaining empty slots as `skippedCount`, break.
  - Reload the plan via `getCurrentPlan()` and return it with `autofillResult`.
  - Note: each `updateSlot` call is its own transaction (matching existing behavior). The overall operation is not wrapped in a single transaction — this is acceptable because partial fills are a valid outcome, and no data is left inconsistent if the process stops mid-way.

**Route (`weekly-plans.routes.js`):**
- Add:
  ```js
  router.post(
    "/current/autofill",
    validate(autofillSchema, "body"),
    (req, res) => {
      const result = weeklyPlansService.autofillEmptySlots(req.body);
      res.json({ data: { ...result.plan, autofillResult: result.autofillResult } });
    },
  );
  ```
- Place this route after `GET /current` and before the `/current/slots/:day` routes (between lines ~35 and ~37 in the current file) for logical grouping of `/current/*` routes.

**Dependencies:** None — uses existing service functions.
**Verification:** Integration tests via Supertest:
- Autofill a fully empty 7-day plan → all 7 slots filled, no duplicates.
- Autofill a plan with 3 filled slots → only 4 empty slots filled, existing slots untouched.
- Autofill with more empty slots than candidate meals → partial fill, `noMoreCandidates: true`.
- Autofill a fully occupied plan → `filledCount: 0`.
- Autofill with `favoritesOnly: true` → only favorites picked.
- Verify notes on empty slots survive autofill.
- Verify `404` when no active plan.

### Frontend Work Package 2: UI Integration

**Scope:** `public/app.js`

**Work:**

1. **Add `autofillPlan()` async function:**
   - Calls `POST /api/weekly-plans/current/autofill` with `{}` (all defaults).
   - On success: updates `state.currentPlan` from `response` (the plan object), re-renders.
   - Reads `autofillResult` from the response to build the status message:
     - `filledCount === 0 && emptyBeforeCount === 0` → "Plan is already full"
     - `filledCount > 0 && skippedCount === 0` → "Filled N empty days"
     - `filledCount > 0 && skippedCount > 0` → "Filled N days — not enough meals for the remaining M"
     - `filledCount === 0 && emptyBeforeCount > 0` → "No meals available to fill empty days"
   - On error: shows error in status banner.

2. **Add "Fill empty days" button to `renderWeeklyPlan()`:**
   - Place in a plan-actions area above the slots (near the stats row), not in the footer (which only shows when meals exist). The button should be visible when the plan exists and has at least one empty slot.
   - Conditional: `const hasEmptySlots = plan.slots.some(s => s.meal === null)`.
   - Render when `hasEmptySlots`:
     ```html
     <button type="button" data-action="plan-autofill" class="secondary">Fill empty days</button>
     ```

3. **Wire the action in `handleWeeklyPlanActions()`:**
   - Add case for `action === "plan-autofill"` that calls `autofillPlan()`.

**Dependencies:** Backend Work Package 1.
**Verification:** Manual testing:
- Create a new empty plan → click "Fill empty days" → all 7 days filled with unique meals.
- Create a plan with 3 days filled → autofill → only 4 empty days filled, originals preserved.
- Seed only 4 meals → create empty plan → autofill → 4 filled, 3 empty, message says not enough meals.
- Fill all 7 days → button disappears (or is hidden).
- Add a note to an empty slot → autofill → note preserved.
