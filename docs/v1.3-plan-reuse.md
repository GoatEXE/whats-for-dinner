# V1.3 Reuse a Past Weekly Plan

## Problem Statement

The weekly planner (v1.2) lets the family plan seven dinners at once, but every new week starts from scratch. In practice, families tend to rotate a relatively stable set of meals — last week's plan is often a great starting point for this week's. Having to re-pick the same meals manually each Sunday adds friction that the planner was supposed to remove.

A lightweight "use this plan as a starting point" action on archived plans gives the family a one-click way to pre-fill a new week from a previous one, then tweak as needed.

## Current Repo Context

Inspected files relevant to this slice:

- **`src/modules/weekly-plans/weekly-plans.repo.js`** — `createPlan(weekStart)` inserts a plan + 7 empty slots in a transaction. `replaceActivePlan(weekStart)` archives the current plan then creates a fresh one. `getPlanById(planId)` returns a full plan with slots, meal summaries, and served status. `listArchivedPlans(limit)` returns archived plan summaries (id, weekStart, createdAt).
- **`src/modules/weekly-plans/weekly-plans.service.js`** — `createPlan(weekStart)` checks for duplicate weekStart against the active plan, then calls either `createPlan` or `replaceActivePlan` on the repo. `getPlanById(planId)` throws 404 for missing plans.
- **`src/modules/weekly-plans/weekly-plans.routes.js`** — `POST /api/weekly-plans` creates a plan from `{ weekStart }`. `GET /api/weekly-plans/history/:id` returns a full archived plan by ID.
- **`src/modules/weekly-plans/weekly-plans.schemas.js`** — `createPlanSchema` validates `weekStart` as a Monday ISO date. `weeklyPlanIdParamSchema` validates `:id` as a positive integer.
- **`src/db/migrations/002_weekly_plans.sql`** — `weekly_plan_slots` has PK `(plan_id, day)` with FK to `meals.id ON DELETE SET NULL`. `idx_weekly_plans_single_active` enforces at most one non-archived plan.
- **`public/app.js`** — `renderPlanHistory()` shows a "Past plans" list with expand/collapse per archived plan via `togglePlanDetail(planId)`. `handleNewWeekPlan()` prompts for a Monday date and calls `createWeeklyPlan(weekStart)`.

## User Flow

1. User expands an archived plan in the "Past plans" section (existing behavior).
2. User sees the plan's slot details (existing behavior) plus a new **"Use as starting point"** button.
3. User clicks the button. A prompt asks for the target Monday date (pre-filled with the next upcoming Monday, matching the existing "New week" prompt behavior).
4. The app creates a new plan for that Monday, pre-filled with the meal assignments and slot notes copied from the source plan. The current active plan (if any) is archived first (existing single-active-plan rule).
5. The new plan appears in the Weekly Plan panel, ready for the user to tweak — swap meals, fill random, clear slots, etc.
6. Served status and history entries are **not** copied. The new plan starts with all slots unserved.

## What Gets Copied

| Source field | Copied? | Rationale |
|---|---|---|
| Slot → `meal_id` | **Yes** | Core value of reuse. If the referenced meal has since been archived, the FK still works — the slot summary shows `isArchived: true` and the user can swap it out. |
| Slot → `notes` | **Yes** | Slot notes like "use leftover chicken" are often reusable context. |
| Plan → `week_start` | **No** | The new plan gets the user-chosen Monday, not the source plan's date. |
| Served status | **No** | Derived from `meal_history` per the v1.2 design. Since the new plan has a different `week_start`, the dates won't match anyway, so slots will naturally show as unserved. |
| History entries | **No** | History belongs to the original plan's dates. |

## In Scope

- New `POST /api/weekly-plans/from/:id` endpoint that creates a plan by copying slots from an archived plan.
- Frontend "Use as starting point" button on expanded archived plan details.
- Reuse of the existing `weekStart` prompt / date-picker pattern.
- Correct interaction with the single-active-plan archive rule.

## Out of Scope

- "Duplicate current plan" (user can just start a new week — the current plan gets archived and can be reused immediately).
- Selecting specific slots to copy (copy all 7, let the user clear unwanted ones).
- Template/saved plan concept (just copy from a real past plan).
- Auto-suggesting the best plan to reuse.
- Editing archived plans.
- Merging two plans together.

## API Design

### `POST /api/weekly-plans/from/:id`

Create a new plan by copying meal assignments and notes from an existing (archived) plan.

**Params:**
- `:id` — the source plan ID (must exist, does not need to be archived — but the main UI flow will only expose this for archived plans).

**Request body:**
```json
{
  "weekStart": "2026-04-13"
}
```

**Rules:**
- `weekStart` must be a valid ISO date falling on a Monday (reuses existing `createPlanSchema` plus `weeklyPlanIdParamSchema`).
- If a non-archived plan already exists for that `weekStart`, return `409 Conflict` (same as `POST /api/weekly-plans`).
- Auto-archives the current active plan for a different week if one exists (same as `POST /api/weekly-plans`).
- The source plan must exist; return `404` if not found.
- Copies `meal_id` and `notes` from each of the source plan's 7 slots into the new plan's corresponding slots.
- Meal references that point to now-deleted rows (`meal_id` set to NULL by `ON DELETE SET NULL`) are copied as-is (NULL), resulting in empty slots — which is correct.

**Response:** `201` with the full new plan object in the standard `{ data: { ... } }` envelope, same shape as `POST /api/weekly-plans` and `GET /api/weekly-plans/current`.

**Example response:**
```json
{
  "data": {
    "id": 4,
    "weekStart": "2026-04-13",
    "isArchived": false,
    "createdAt": "2026-04-06T18:30:00.000",
    "updatedAt": "2026-04-06T18:30:00.000",
    "slots": [
      {
        "day": 0,
        "date": "2026-04-13",
        "label": "Monday",
        "meal": { "id": 3, "name": "Spaghetti Tacos", "isFavorite": true, "isArchived": false },
        "notes": null,
        "served": false
      },
      {
        "day": 1,
        "date": "2026-04-14",
        "label": "Tuesday",
        "meal": null,
        "notes": null,
        "served": false
      }
    ]
  }
}
```

Note: all `served` fields are `false` because the new plan's dates have no matching `meal_history` entries.

## Interaction with Existing Plan Behavior

### Single-active-plan rule

The new endpoint follows the same archive-on-create logic as `POST /api/weekly-plans`:
- If no active plan exists → create directly.
- If an active plan exists for a *different* `weekStart` → archive it, then create.
- If an active plan exists for the *same* `weekStart` → `409 Conflict`.

This is already implemented in `weekly-plans.service.js#createPlan`. The new service function should reuse the same checks and delegate to a repo function that extends the create-plan transaction with slot copying.

### Archived plan detail

The source plan is fetched via `weeklyPlansRepo.getPlanById(id)`, which already resolves full slot data. But for the copy operation, we only need the raw `meal_id` and `notes` per slot — not the computed meal summaries or served status. The repo should read slots directly from `weekly_plan_slots` for efficiency, not go through `buildPlan`.

### Shopping list

No interaction. After the new plan is created with copied slots, the "Generate shopping list from plan" button works as before.

### History

No interaction. Served status is derived from `meal_history` joined on `(meal_id, served_on)` where the date is computed from the new plan's `weekStart`. Since the dates differ from the source plan, nothing carries over.

### Archived meals in copied slots

A copied slot may reference a meal that has been soft-archived since the source plan was active. The existing plan rendering already handles this: the meal summary includes `isArchived: true` and the UI shows an "(Archived)" badge. The user can clear the slot and pick a replacement. No special handling needed.

## Data and Storage Implications

- **No migration needed.** The existing `weekly_plans` and `weekly_plan_slots` tables are sufficient. The copy operation is a standard insert of a new plan row + 7 slot rows.
- **No new tables or columns.** The source plan ID is not persisted on the new plan — this is a stateless copy operation, not a template relationship.

## Acceptance Criteria

### Backend

1. `POST /api/weekly-plans/from/:id` with a valid source plan ID and Monday `weekStart` creates a new plan and returns `201`.
2. The new plan's slots have the same `meal_id` and `notes` values as the source plan's slots.
3. All slots in the response have `served: false`.
4. If the source plan ID does not exist, return `404`.
5. If a non-archived plan already exists for the given `weekStart`, return `409`.
6. If an active plan exists for a different week, it is archived before the new plan is created.
7. If a source slot's `meal_id` references a soft-archived meal, the slot is still copied with that meal ID and the response shows `isArchived: true` in the meal summary.
8. If a source slot's `meal_id` is NULL (empty slot or hard-deleted meal), the corresponding new slot is also NULL.
9. The response envelope and plan shape match exactly the existing `POST /api/weekly-plans` response.

### Frontend

1. The expanded archived plan detail shows a "Use as starting point" button.
2. Clicking the button prompts for a Monday date (pre-filled with the next upcoming Monday).
3. After confirmation, the new plan is created and the weekly plan panel updates to show it.
4. The past-plans list refreshes to include the newly archived plan (if one was displaced).
5. If the request fails (e.g. 409 conflict), the error is shown in the status banner.

## Risks and Assumptions

| Risk / Assumption | Impact | Mitigation |
|---|---|---|
| Source plan may have many stale/archived meal references after weeks pass. | Low — user sees "(Archived)" badges and can clear/replace. | The UI already handles `isArchived: true` meal summaries. No extra work. |
| User might accidentally overwrite their current plan. | Medium — the current plan gets archived, which is recoverable. | The prompt step gives the user a chance to cancel. The archived plan remains accessible in "Past plans." Consider a confirmation if an active plan exists. |
| No link between the new plan and its source means you can't trace provenance. | Low — this is a family dinner tool, not an audit system. | Acceptable for v1.3. A `source_plan_id` column could be added later if needed, but adds no user value now. |
| The `idx_weekly_plans_single_active` unique partial index prevents two non-archived plans. | None — this is desired. The create flow archives first, then inserts. | Already tested and working in v1.2 implementation. |

## Implementation Tasks

### Backend Work Package 1: Repo and Service

**Scope:** `src/modules/weekly-plans/weekly-plans.repo.js`, `src/modules/weekly-plans/weekly-plans.service.js`

**Work:**

**Repo (`weekly-plans.repo.js`):**
- Add a `listSlotAssignments(planId)` function that returns raw `{ day, mealId, notes }` rows from `weekly_plan_slots` for a given plan ID. This avoids the overhead of `buildPlan` (which resolves meal summaries and served status) when we only need the slot data for copying.
- Add a `createPlanFromSource(weekStart, sourceSlots)` function:
  - Inserts a new `weekly_plans` row.
  - Inserts 7 `weekly_plan_slots` rows, using `meal_id` and `notes` from the `sourceSlots` array.
  - Runs in a single transaction.
  - Returns the new plan ID.
- Add a `replaceActivePlanFromSource(weekStart, sourceSlots)` function:
  - Archives all active plans (reuse `archiveActivePlansStatement`), then calls `createPlanFromSource`.
  - Runs in a single transaction.
  - Returns the new plan ID.

These mirror the existing `createPlan` / `replaceActivePlan` pair but accept slot data instead of creating empty slots.

**Service (`weekly-plans.service.js`):**
- Add a `createPlanFromSource(sourcePlanId, weekStart)` function:
  - Validates the source plan exists via `weeklyPlansRepo.getPlanById(sourcePlanId)` — throw `404` if null.
  - Checks for weekStart conflict with the active plan (same logic as existing `createPlan`): same weekStart → `409`; different weekStart → archive + create; no active plan → create directly.
  - Reads source slot assignments via `weeklyPlansRepo.listSlotAssignments(sourcePlanId)`.
  - Delegates to `createPlanFromSource` or `replaceActivePlanFromSource` on the repo.
  - Returns the full new plan via `getPlanById`.
  - Wraps SQLite unique constraint errors as `409` (same pattern as existing `createPlan`).

**Dependencies:** None — extends existing module.
**Verification:** Unit tests for `listSlotAssignments` and integration tests for the full create-from-source flow.

### Backend Work Package 2: Route and Schema

**Scope:** `src/modules/weekly-plans/weekly-plans.routes.js`, `src/modules/weekly-plans/weekly-plans.schemas.js`

**Work:**

**Route (`weekly-plans.routes.js`):**
- Add `POST /from/:id` route:
  ```js
  router.post(
    "/from/:id",
    validate(weeklyPlanIdParamSchema, "params"),
    validate(createPlanSchema, "body"),
    (req, res) => {
      const plan = weeklyPlansService.createPlanFromSource(
        req.params.id,
        req.body.weekStart,
      );
      res.status(201).json({ data: plan });
    },
  );
  ```
- The existing `weeklyPlanIdParamSchema` and `createPlanSchema` are sufficient — no new schemas needed.

**Schema (`weekly-plans.schemas.js`):**
- No changes needed. The existing schemas cover the param and body validation.

**Dependencies:** Backend Work Package 1.
**Verification:** Integration tests via Supertest:
- Copy from a populated archived plan → new plan has correct meals/notes, all unserved.
- Copy from a plan with empty slots → empty slots copy as empty.
- Copy from a plan with an archived meal → slot has meal with `isArchived: true`.
- Source plan not found → `404`.
- Duplicate weekStart → `409`.
- Active plan archived on create → verify old plan appears in history.
